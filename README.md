_Разработка графического пользовательского интерфейса (GUI) для языкового процессора_

### Цель работы
Разработать оконное приложение - текстовый редактор с графическим интерфейсом пользователя, который в дальнейшем может быть расширен до полноценного языкового процессора для анализа исходного кода.

### Описание проекта
В рамках лабораторной работы реализовано настольное приложение «Text editor» - специализированный текстовый редактор с графическим интерфейсом пользователя. Программа предназначена для создания, открытия, редактирования и сохранения текстовых файлов, а также содержит отдельную область для вывода результатов работы языкового процессора.

Интерфейс приложения состоит из основного меню, панели инструментов, области редактирования текста и области отображения результатов. Пользователь может изменять размеры окна приложения и соотношение размеров рабочих областей.

На текущем этапе реализован базовый функционал текстового редактора. Пункты меню «Текст» и команда «Пуск» предусмотрены архитектурой программы и будут реализованы на следующих этапах разработки.

### Диагностика и нейтрализация синтаксических ошибок
В соответствии с требованиями лабораторной работы применён метод Айронса. 
Основная идея метода – при обнаружении несоответствия текущего токена ожидаемой
конструкции анализатор не прекращает работу а отрабатывает до конца по каждому токену


Парсер заранее хранит три эталонных шаблона корректного оператора объявления целочисленной переменной: 
базовый (final int идентификатор = число ; ), с унарным плюсом и с унарным минусом. 
При разборе каждой строки анализатор последовательно сравнивает полученные лексемы с 
каждым шаблоном

Если текущий токен соответствует ожидаемому символу, разбор движется вперёд. 
Если нет – метод Айронса предлагает: закинуть его в ошибку
и продолжить сравнение с тем же ожидаемым символом

Разбор ведётся построчно – каждый оператор выделяется до
точки с запятой или до перехода на новую строку. 
Это локализует последствия ошибки: неудачное восстановление 
в одной строке не влияет на анализ последующих. 
После обработки текущего оператора парсер перемещается к следующему фрагменту 
кода и повторяет процедуру. Таким образом, даже при наличии множества 
синтаксических дефектов программа не останавливается, а доходит до конца входного файла,
выдавая полный список диагностических сообщений для каждой ошибочной конструкции.

#### Задача 2. 
Построить РВ для валидации российских номеров мобильных телефонов\
(начинаются с цифры 8 символов +7, далее зоновый код в скобках или без и номер абонента, \
номер абонента может быть разделен пробелами или символами -).

Описание задачи:
Найти телефонные номера российского формата, начинающиеся с 8 или +7, с возможными разделителями (пробел, дефис, круглые скобки).

Регулярное выражение:

(([+][7])|[8])([(]||[ ]|[-])\d{3}([)]||[ ]|[-])\d{3}(||[ ]|[-])\d{2}(||[ ]|[-])\d{2}$

Примеры строк, которые должны находиться:

8(913)555-12-34\
+7 913 555 12 34\
8-913-555-12-34

Примеры строк, которые не должны находиться:

+8(913)555-12-34
7(913)555-12-34
8(913)555-1234

#### Задача 3. 
Построить РВ, описывающее IP-адрес (v4) с маской подсети. \
Учесть корректные значения, например, порты находятся в диапазоне от 0 до 65535.

Описание задачи:
Найти подстроки вида IPv4/маска:порт, где:

каждый октет IPv4 находится в диапазоне 0–255;
маска подсети — от 1 до 32;
порт — от 0 до 65535.

Регулярное выражение:

(\d|[1]\d{1,2}|[2][0-4]\d|[2][5][0-5])[.]
(\d|[1]\d{1,2}|[2][0-4]\d|[2][5][0-5])[.]
(\d|[1]\d{1,2}|[2][0-4]\d|[2][5][0-5])[.]
(\d|[1]\d{1,2}|[2][0-4]\d|[2][5][0-5])(\/)(\d|[1-2]\d|[3][1-2])[:]
